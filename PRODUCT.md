# Product

## Register

product

## Users

QuickBid is used by hospital IT bid teams, pre-sales engineers, bid document specialists, and reviewers who need to turn a tender file into a compliant bid package. They work under deadline pressure, need to inspect extracted requirements, correct AI output, reuse existing materials, and keep evidence traceable back to the tender document.

## Product Purpose

QuickBid helps users produce hospital-IT bid documents through a confirmation-driven multi-agent workflow. Parser, Matcher, Generator, Reviewer, SubBid, and QA agents each perform a focused step, then wait for user confirmation or correction before the project advances. Success means users can upload a tender, validate structured requirements, shape the outline, match materials, generate commercial and technical bid content, review compliance issues, and export deliverables with clear source evidence.

## Brand Personality

Practical, precise, calm.

The interface should feel like a focused professional workbench: dense enough for real bid work, explicit about state and evidence, and restrained in visual treatment. It should prioritize clarity, correction, and auditability over decorative presentation.

## Anti-references

Avoid marketing-style hero layouts, oversized decorative cards, vague AI assistant theatrics, and workflows that hide the current project state. Avoid making users memorize magic phrases when a natural instruction can be mapped to a safe tool action. Avoid review output that only reports generic warnings without showing the tender requirement, generated document location, and concrete fix.

## Design Principles

1. Evidence before confidence: extracted facts, risks, outline items, matches, and review findings should show source pages or field paths whenever possible.
2. State is part of the interface: every tool action must make clear what project data exists, what is missing, and what can run next.
3. Confirmable automation: agents should do useful work independently, but user correction remains a first-class path.
4. Workbench density: use compact, consistent product UI patterns suited to repeated inspection, comparison, and correction.
5. Separate bid volumes early: commercial, technical, and price content should diverge from the outline stage rather than being split only at export time.

## Accessibility & Inclusion

Default to WCAG AA contrast for text and controls. Keep keyboard access for chat actions, report tabs, file preview close, and export actions. Motion should be short and state-driven, with no decorative page-load choreography. Status should not rely on color alone; pair color with labels, icons, and counts.
